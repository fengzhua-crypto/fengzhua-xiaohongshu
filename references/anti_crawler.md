# 小红书反爬策略与应对方案

## 小红书反爬机制概览

小红书（XHS）有多层反爬体系，以下是主要机制及本工具的应对方式。

### 1. 登录验证

**机制**：大部分内容需要登录后才能查看，未登录状态下搜索结果和笔记详情受限。

**应对**：
- 本工具采用「手动登录 + Cookie 持久化」方案
- 首次运行 `init` 命令时，打开真实浏览器让用户手动扫码登录
- 登录后将 cookies 和 localStorage 保存到本地（`~/.xhs_scraper/storage_state.json`）
- 后续采集自动加载会话，无需重复登录
- Cookie 有效期通常为 7-30 天，过期后重新执行 `init` 即可

### 2. 浏览器指纹检测

**机制**：小红书通过 `navigator.webdriver`、`navigator.plugins` 等属性检测自动化工具。

**应对**：
- 使用 Playwright 启动真实 Chromium 浏览器（非无头 HTTP 请求）
- 注入反检测脚本，隐藏 `webdriver` 标识
- 模拟真实的 `plugins`、`languages`、`chrome` 对象
- 启动参数添加 `--disable-blink-features=AutomationControlled`

### 3. 请求签名（X-s / X-t）

**机制**：小红书 API 请求需要携带 `X-s` 和 `X-t` 签名参数，由前端 JS 生成。

**应对**：
- 本工具不直接调用 API，而是通过浏览器渲染页面后提取 DOM 内容
- 浏览器会自动生成签名，无需手动计算
- 如果未来需要调用 API，可考虑使用 `page.evaluate()` 在浏览器环境中执行签名函数

### 4. 频率限制

**机制**：短时间内大量请求会触发风控，导致：
- 页面加载变慢
- 验证码弹出
- 账号临时封禁

**应对**：
- 采集间隔随机化（1.5-3.5 秒）
- 模拟人类滚动行为（随机滚动距离和等待时间）
- 批量采集笔记详情时，每条间隔 2-4 秒
- 单次采集量建议不超过 50 条
- 如遇验证码，脚本会暂停，可手动处理

### 5. 设备指纹

**机制**：小红书通过 Canvas、WebGL 等生成设备指纹。

**应对**：
- 使用真实浏览器（Playwright + Chromium），Canvas/WebGL 指纹为真实值
- 每次启动使用相同的 User-Agent 和 viewport，保持一致性
- 可考虑使用 `--persistent` 模式保持浏览器配置文件

## 最佳实践

### 采集频率建议

| 场景 | 建议频率 | 单次量级 |
|------|----------|----------|
| 关键词搜索 | 每小时不超过 5 次 | ≤ 20 条/次 |
| 用户主页采集 | 每天不超过 10 个用户 | ≤ 20 条/用户 |
| 笔记详情 | 每分钟不超过 10 条 | - |
| 批量详情采集 | 每小时不超过 100 条 | - |

### 风控信号

如果出现以下情况，立即停止采集：

1. **验证码频繁弹出** — 说明已被风控，暂停至少 2 小时
2. **搜索结果为空但关键词有效** — 可能被临时限制
3. **页面加载极慢** — 可能被降速
4. **笔记详情页显示「内容不存在」** — 笔记被删除或被限制访问

### Cookie 续期

Cookie 过期后会出现：
- 搜索结果为空
- 笔记详情页跳转到登录页
- 页面提示「请登录」

解决方法：重新执行 `python xhs_scraper.py init`

## 法律与合规提示

1. **仅用于个人研究分析**：采集的数据应用于市场调研、竞品分析等合法用途
2. **不批量搬运内容**：不将他人原创内容直接搬运到自己的账号
3. **尊重版权**：图片、文字版权归原作者所有
4. **遵守 Robots 协议**：关注小红书 robots.txt 的变更
5. **控制采集量**：不对平台服务造成压力
6. **不绕过付费内容**：不采集需要付费才能查看的内容
